Functional Requirements of the Simulation Manager

This page lists and describes the functional requirements of the EODiSP model manager. The model manager is responsible to manage the federates running on a node.

Contents

  • Purpose of the model manager
  • Functional Requirements
  • Simulation Manager Runner

Purpose of the model manager

The basic functionality of the model manager is to manage a repository of federates that can be started on request. The model manager does not directly host the federeates but runs them in their own virtual machine, i.e. in the Local RTI component. An overview of the model manager's funcitonality is given in the list below:

  • Starting a federate application
  • Kill a federate application
  • Provide a list of running federate applications
  • Provide information about running federates

Functional Requirements

The following table is a list of detailed functional requirements. These requirements are not explicitely covered by the user requirements, but are implicitely needed. The table lists them more formally and in more detail. This requirements will be implemented by the core implementation of the EODiSP framework. The code for the repository resides in the core package (EODiSP core).

The table is organised as follows:

  • Column 1: A consecutive number identifying the functional requirement.
  • Column 2: A detailed description of the functional requirement.
Nr. Name Description
mm_01 Starting a federate

Starting an application means to create a new virtual machine which hosts one or more federates. The model manager needs to make sure that the appropriate configuration is delivered to the virtual machine (Local RTI Component settings). All configuration parameters are provided by the simulation manager and include at least the following:

  • Network address (InetAddress ?) and id of the CRC
  • The id of the application to be started
  • Name of the federation execution the federates shall join (some federates may not need this). It is not supported to specify the federation execution name for each federate in one application individually.
mm_02 Kill a federate application

Forcibly kills the JVM that runs the application hosting the federates. The only parameter needed is the id of the application.

mm_03 Provide a list of running federates

Provides the following information for each running federates:

  • Id of the federate
  • The network address of the CRC the federate is joined to
  • Optional: The status of the federate (active/passive). Only possible if the federate implements this functionality according to the EODiSP Federate specification (TBD: link to spec).
