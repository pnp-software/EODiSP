Developer Manuals

This page lists and describes the functional requirements of the EODiSP repository. The EODiSP repository is used to keep track of all simulation manager applications as well as all federates that participate in a simulation network.

Purpose of the repository

The idea behind the EODiSP respository is to have a place where all simulation and model manager applications as well as all federates can register themeselves. The need for such a repository stems from the fact that, in a distributed environment, it is almost impossible to assemble a simulation (e.g. a federation) without offline informations about the federates. Furthermore, the EODiSP requirements ask for a a facility to limit access to federates that are included in a model manager. This means, only some simulation manager applications are allowed to access certain federates. This ensures that the model owner can control who may access his models.

The following list summarises the basic idea behind the EODiSP repository:

  • A model owner shall have control over the access to his models. He can achieve this by retrieving a list of registered simulation manager applications from the repository and allow/deny access for a certain model for a certain simulation manager application.
  • As a consequence to the previous requirement, a simulation manager application needs to be able to register itself in the repository.
  • A simulation manager application shall be able to retrieve a list of all federates that are defined in the current simulation network in order to assemble a federation. This has the advantage that federates do not need to be running at the time of federation assembling.
  • Every federate has some information attached to it. These informations can be used to access the federate within the simulation network. Such informations are stored in the repository along with the definition of the federate itself.
  • Every federate shall specify to which federation it will eventually participate. This makes it possible to validate a federation prior to its start. Therefore, the repository shall store a list of available federations to which federates can contribute.

Installation Requirements

The basic idea behind the repository is to store informations about various parts in a simulation network without the need of having these parts running all the time. This, however, implies that these offline informations must be available at any time for any participant in the simulation network. Therefore, the repository must be running all the time, preferrably on a publicly accessible server. This ensures the availability of the offline data for all participant in the simulation network. Other from that, there are no specific requirements for the repository. The remote connection to/from the repository will be handled by the EODiSP remote package.

Functional Requirements

The following table is a list of detailed functional requirements. These requirements are not explicitely covered by the user requirements, but are implicitely needed. The table lists them more formally and in more detail. This requirements will be implemented by the core implementation of the EODiSP framework. The code for the repository resides in the core package (EODiSP core).

The table is organised as follows:

  • Column 1: A consecutive number identifying the functional requirement.
  • Column 2: A short description of the functional requirement. It is formulated as a request that should be handled.
  • Column 3: A detailed description of the functional requirement..
Nr. Short Description Description
rep_01 Register a simulation manager application. Every simulation manager application needs to register itself in the repository. The following informations are stored for each:
  • Name of the application as a string.
  • Name of the person in charge of the application (optional)
  • Telephon number of the person in charge of the application (optional)
  • E-Mail of the person in charge of the application (optional)
  • Country where the application runs as string.
  • A unique identifier of the simulation manager application as a guid.
  • An identifier defining the location of the simulation manager application in the simulation network as an INET address.
rep_02 Unregister a simualtion manager application.

A simulation manager application that has been registered in the repository can unregister itself. This function is only available for simualtion manager applications that are already registered, otherwise it is ignored.

Identification of a simulation manager application is done through the unique identifier (guid) of the application. This is stored in the repository and must match the identifier of the unregister request.

Whenever a valid unregister request arrives at the repository, all repository data (see rep_01) belonging to the respective simulation manager application will be deleted.

rep_03 Update a simulation manager application.

Some, but no all, of the date in the repository can be updated. This can be necessary in case that some properties change. The unique identifier will be used to identify the simulation manager application for which the data should be updated. In case of valid update request, the following data may be changed:

  • Name of the simulation manager application as a string.
  • Name of the person in charge of the simulation manager application (optional)
  • Telephon number of the person in charge of the simulation manager application (optional)
  • E-Mail of the person in charge of the simulation manager application (optional)
  • An identifier defining the location of the simulation manager application in the simulation network as an INET address.

All other data will be static and cannot be changed with an update request.
rep_04 Delete a simulation manager application.

In some cases, it might be useful to be able to delete simulation manager application entries in the repository. For instance, if a simulation manager application installation is removed, there is no way to unregister the application from the repository. The delete request, however, enables the user to delete the appropriate entries.

The request can be done with the repository manager application (through the GUI). To identify the entry which should be deleted, the user is presented with a list of registered simulation manager applications from which he can choose from. Once an entry is selected, a deregister request will be executed in order to delete the appropriate entry.

Allowing this request is dangerous in a sense that all entries in the repository can be deleted. However, as long as no problem (and no requirements) in this direction occur, this request can be done without restriction. At a later stage, it might be possible to extend it to only allow this request for registered user (by querying for a password or similar).

rep_05 Registration of model manager application.

The functionality is the same as for the simulation manager application (see rep_01 - rep04). The following functionality is provided:

  • Register model manager application from the repository
  • Unregister model manager application from the repository
  • Update model manager application in the repository
  • Remove model manager application from the repository
rep_05 Create a federate category.

A federate that is registered in the repository can be categorized. Each category can have zero to many federates that are logically grouped. Usually, this means that a category represents one federation execution. As a consequence, federates in a category should work together in a sense that the form a complete federation execution, where the outputs of one federate are consumed by another federate and so on.

The following informations are stored with each category:

  • The name of the category as string.
  • A list of SOM's that are linked with this category.
  • A boolean value indicating whether a category is closed or open. In a closed category, no additional federates can be added.

Categories can be created in the repository manager application (through the GUI application). If a category already exists, this request will be ignored, otherwise, a new repository will be created. Repositories are organized in a flat dimension (as opposed to a hierachical organization).

rep_06 Update the federate category.

The information stored along with a category in the repository can be changed. The following list indicates which values can be changed with an update request:

  • The name of the category
  • The boolean value indicating whether a category is closed or open.

If the name is changed to a value which is already defined in the repository, the request will be ignored. A change request can be performed with the repository manager application (through the GUI application).

The list of SOM's that are linked to the repository (see rep_05) is updated automatically whenever a SOM is added to the category. Therfore, it is not needed to update this values manually.

rep_07 Delete a federate category.

An existing federate category in the repository can be deleted. All links to SOM's will be lost when a category is deleted. All informations about the category will be deleted as well (without backup).

This request can be performed with the repository manager application (through the GUI application).

rep_08 Add a SOM to the repository.

A SOM (Simulation Object Model) can be added to the repository. A SOM defines the in- and output of a federate. It can be either added to no, one or more categories. It is possible that several federates comply to a single SOM (i.e. implementing the interface defined by the SOM). If the same SOM is already stored in the repository, the request will be ignored.

The SOM itself is complient with the IEEE HLA specification and is not modified by the repository. The values that are stored within a SOM can therefore defined by the standard.

Since the SOM itself needs some metadata, the SOM will be saved along with additional data. The following data is stored in such a SOM object:

  • A unique Id identifying the SOM.
  • A link the SOM on the local file path.

This request can be performed with the model manager application.
rep_09 Delete a SOM from the repository

A SOM can be deleted from the repository. All links to this SOM from any federate are deleted when this request is performed.

This request can be performed with the repository manager (through the GUI application).

rep_10 Add a federate to the repository.

A federate can be added to the repository. In the repository, not the federate itself (i.e. the executable code) is stored, but just additional informations about it. The following informations are stored along with the federate

  • The name of the federate.
  • A description of the federate.
  • A unique identifier for the federate as a guid.
  • An identifier defining the location of the federate in the simulation network as an INET address.
  • An Id of the target executable this federate will be run in.
  • An Id of one or more SOM's to which this federate complies
  • A list of ID's that uniqeuly identifying simulation manager applications that have access to this federate. This prevents simulation manager applications without given access to the model from displaying this federate..
  • The Id of the model manager that holds this federate.
  • A link to the init data description.

The list of simulation manager application ID's will only used to filter the display in the GUI application of the simulation manager application. It is not actual implementation of the security. The implementation of the model manager itself will ensure that only simulation manager applications that have proper access to a federate are allowed to interact with them.

This request can be performed with the model manager application.
rep_11 Update a federate in the repository.

A federate that is already stored in the repository can be updated. This can be the case if the implementation of a federate has changed or the if the model owner wants to change the data associated with the federate. The following data can be changed in the repository:

  • The name of the federate.
  • The description of the federate
  • The identifier defining the location of the federate in the simulation network (the INET address.
  • The Id of the target executable this federate will be run in.
  • Links to SOM's can be added or removed.
  • The list of ID's that uniqeuly identifying simulation manager applications that have access to this federate.

This request can be performed with the model manager application (through the GUI application).

rep_12 Delete a federate from the repository.

A federate can be deleted from the repository. The link to the SOM will be deleted, if such a link exists for the federate. All information about the federate currently stored in the repository will be lost.

rep_13 Create a backup of the repository.

In order to prevent loss of all data, a backup can be created. The backup will store all data currently stored in the repository and will crate a file with a unique name (preferrably with the current date/time in its name).

This request can be performed with the repository manager application (through the GUI application).

rep_14 Restore the repository from a backup.

The data in the repository can be restored by using a backup file that has already been created. All data currently stored in the repository will be deleted and replaced by the data stored in the backup. No merge in any sense will be performed.

This request can be performed with the repository manager application (through the GUI application).

Repository History

The repository stores data that can be changed. In general, it would be possible to store this changes in a history from where changes could be redone, viewed, etc. This functionality, however, is not implemented in the EODiSP respository.

Transactions

Since the EODiSP respository can be accessed from different sources (e.g. simulation/model manager applications), it must provided the means to ensure that there are no problems related to synchronization.
